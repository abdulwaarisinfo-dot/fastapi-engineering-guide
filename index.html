01

What is FastAPI?

The Simple Idea

Imagine a highway with no traffic lights. Regular Python web servers stop at every intersection waiting for a car to pass. FastAPI builds highways where multiple cars move simultaneously — requests come in, get processed without waiting for each other, and responses fly back at full speed. It is the fastest Python web framework for building APIs, and it comes with automatic documentation built right in.

FastAPI is a modern, high-performance Python web framework for building RESTful and GraphQL APIs, based on standard Python type hints. Built by Sebastián Ramírez and released in 2018, it sits on top of Starlette (the ASGI toolkit) and Pydantic (the data validation engine), combining both into a single coherent development experience.

It is the dominant Python framework for AI-backend infrastructure, replacing Flask in greenfield production systems due to its native async support, automatic OpenAPI schema generation, and first-class Pydantic integration that eliminates entire categories of boilerplate validation code.

"FastAPI is not just fast in terms of runtime throughput — it is fast to develop with. Automatic docs, type-driven validation, and zero-config dependency injection compress weeks of backend work into days."

ASGI-Native

Non-blocking by design

Built on Starlette's ASGI core, every endpoint can be async def — handling thousands of concurrent connections without spawning extra threads or processes.

Type-Driven Validation

Declare types, get validation free

Python type annotations are not just documentation — FastAPI parses them at startup and uses Pydantic to validate every request body, query parameter, and path variable automatically.

Auto Documentation

Swagger UI generated for free

On every app start, FastAPI introspects routes and models to generate a live OpenAPI 3.0 spec, Swagger UI at /docs, and ReDoc at /redoc — with no extra configuration.

Dependency Injection

Wiring handled by the framework

FastAPI's DI system resolves auth tokens, database sessions, config objects, and nested dependencies declaratively — no service locators, no manual wiring, no hidden global state.

Pydantic v2 Core

Rust-speed validation in Python

Pydantic v2 (used in FastAPI 0.100+) rewrites the validation core in Rust, providing 5–50x faster model instantiation and serialization compared to Pydantic v1 — directly improving request/response throughput.

Production Performance

Benchmarks at the top of Python frameworks

In TechEmpower benchmarks, FastAPI consistently ranks among the top Python frameworks — within 2–3x of Go and Node.js performance for I/O-bound workloads, making it viable for high-traffic production APIs.

FastAPI positions itself against two legacy incumbents. Flask is minimal but synchronous-first, requiring extensions for everything from validation to async. Django REST Framework provides batteries but enforces Django's ORM-centric patterns and adds significant overhead for pure API services. FastAPI occupies the ideal middle ground: production features out of the box, async-first, no legacy baggage.

Core Architecture & ASGI Internals

The Simple Idea

Old web servers handle one customer at a time — they take your order, walk to the kitchen, wait for the food, bring it back, then serve the next customer. Async servers are like waiters that take ten orders simultaneously, drop them all in the kitchen, and pick up whatever is ready first. ASGI is the protocol that makes this possible in Python.

FastAPI's performance stems from its ASGI (Asynchronous Server Gateway Interface) foundation. ASGI replaced WSGI as Python's web server standard, adding native support for long-lived connections, WebSockets, and non-blocking I/O through Python's asyncio event loop.

ASGI Request Lifecycle

↓

↓

↓

↓

↓

The event loop is the beating heart of async Python. A single-threaded event loop can juggle thousands of concurrent I/O-bound operations — database queries, HTTP calls, file reads — by suspending at await points and resuming when data is ready. CPU-bound tasks still block the loop and should be offloaded to asyncio.run_in_executor or a process pool.

Sync vs Async — critical behavioral difference PYTHON 
from fastapi import FastAPI
import asyncio, time

app = FastAPI()

# SYNC: blocks the entire event loop for 2 seconds
# 100 concurrent requests → 200 seconds total wait
@app.get("/sync")
def sync_endpoint():
    time.sleep(2)        # BLOCKS — entire server freezes
    return {"status": "done"}

# ASYNC: yields control during await
# 100 concurrent requests → ~2 seconds total wait
@app.get("/async")
async def async_endpoint():
    await asyncio.sleep(2)  # yields — other requests proceed
    return {"status": "done"}
Production Note

Never call blocking I/O (requests, pymongo, time.sleep, open()) inside async def endpoints. It blocks the entire event loop. Use motor for MongoDB, httpx for HTTP calls, and aiofiles for file I/O — all of which are non-blocking.

Routing, Path Parameters & Query Parameters

The Simple Idea

Routing is just the address system of your API. /users/42 means "give me user number 42" — that 42 is a path parameter. /users?role=admin&page=2 is filtering: those are query parameters. FastAPI reads your function signature and extracts both automatically from the URL.

Complete routing patterns — GET, POST, PUT, PATCH, DELETE PYTHON 
from fastapi import FastAPI, APIRouter, Path, Query, HTTPException
from typing import Optional

router = APIRouter(prefix="/users", tags=["Users"])

# GET — retrieve single resource by ID
@router.get("/{user_id}")
async def get_user(
    user_id: str = Path(..., description="MongoDB ObjectId"),
):
    return {"id": user_id}

# GET — list with query filters + pagination
@router.get("/")
async def list_users(
    role: Optional[str] = Query(None, description="Filter by role"),
    page: int = Query(1, ge=1),
    limit: int = Query(20, le=100),
):
    skip = (page - 1) * limit
    return {"page": page, "limit": limit, "skip": skip}

# POST — create resource
@router.post("/", status_code=201)
async def create_user(payload: UserCreate):
    return {"created": payload.email}

# PUT — full replacement
@router.put("/{user_id}")
async def replace_user(user_id: str, payload: UserUpdate):
    return {"updated": user_id}

# PATCH — partial update
@router.patch("/{user_id}")
async def patch_user(user_id: str, payload: UserPatch):
    data = payload.model_dump(exclude_unset=True)  # only provided fields
    return {"patched": user_id, "fields": list(data.keys())}

# DELETE
@router.delete("/{user_id}", status_code=204)
async def delete_user(user_id: str):
    return None  # 204 No Content returns no body

Use APIRouter to modularize routes by domain — one router per resource type. Register routers on the app at startup, applying shared prefixes and tags. This keeps the main app instance clean and each domain file independently testable.

05

Request Body, Pydantic & Data Validation

The Simple Idea

Pydantic is a strict contract for your data. You declare what shape an object should be — its fields, types, and constraints — and Pydantic rejects anything that doesn't match before your code ever runs. Think of it as a bouncer at the API door: bad data never gets past the entrance.

Production Pydantic models — User domain PYTHON 
from pydantic import BaseModel, EmailStr, Field, field_validator
from typing import Optional
from datetime import datetime
import re

class UserCreate(BaseModel):
    email: EmailStr
    username: str = Field(..., min_length=3, max_length=50, pattern=r"^[a-zA-Z0-9_-]+$")
    password: str = Field(..., min_length=8)
    full_name: Optional[str] = Field(None, max_length=100)

    @field_validator("password")
    @classmethod
    def validate_password_strength(cls, v: str) -> str:
        if not re.search(r"[A-Z]", v):
            raise ValueError("Password must contain an uppercase letter")
        if not re.search(r"\d", v):
            raise ValueError("Password must contain a digit")
        return v

class UserResponse(BaseModel):
    # Never expose: password_hash, internal_flags
    id: str
    email: EmailStr
    username: str
    full_name: Optional[str]
    created_at: datetime

class UserPatch(BaseModel):
    # All fields optional for PATCH semantics
    full_name: Optional[str] = None
    username: Optional[str] = None

A critical production pattern: always define separate Create, Response, and Update models per entity. The Create model accepts raw input including passwords. The Response model explicitly whitelists fields returned to clients — never exposing password hashes or internal fields. Using model_dump(exclude_unset=True) on PATCH payloads ensures only explicitly provided fields get written to the database.

Dependency Injection System

The Simple Idea

Instead of every function going out to get its own tools — opening its own database connection, decoding its own JWT token — FastAPI hands them pre-assembled. You declare what your function needs, and FastAPI delivers it. Shared resources get created once and passed to whoever needs them. No global variables, no tight coupling, no repetition.

Dependency graph — database session + authenticated user PYTHON 
from fastapi import Depends, HTTPException, status
from motor.motor_asyncio import AsyncIOMotorDatabase
from app.database import get_database
from app.utils.auth import decode_access_token

# Layer 1: database session injected into every route that needs it
async def get_db() -> AsyncIOMotorDatabase:
    return get_database()  # returns shared Motor client

# Layer 2: auth — decodes JWT, returns user_id or raises 401
async def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: AsyncIOMotorDatabase = Depends(get_db),
) -> dict:
    payload = decode_access_token(token)  # raises if invalid
    user = await db.users.find_one({"_id": payload["sub"]})
    if not user:
        raise HTTPException(status_code=401, detail="User not found")
    return user

# Layer 3: admin check — depends on auth
async def require_admin(current_user: dict = Depends(get_current_user)):
    if current_user.get("role") != "admin":
        raise HTTPException(status_code=403, detail="Admin access required")
    return current_user

# Usage — clean, composable, testable
@router.delete("/users/{user_id}")
async def admin_delete_user(
    user_id: str,
    admin: dict = Depends(require_admin),   # pulls in auth + db
    db: AsyncIOMotorDatabase = Depends(get_db),
):
    await db.users.delete_one({"_id": user_id})
    return {"deleted": user_id}

Dependencies are cached within a request by default: if three route functions in a single request each declare Depends(get_db), FastAPI calls get_db() once and reuses the result. This makes connection pooling predictable and eliminates duplicate work within a single request-response cycle.

07

Authentication & JWT Security

The Simple Idea

JWT is like a signed stamp on a wristband at a concert. The venue (your server) stamps it once at login. Every time you enter a different area (hit an endpoint), bouncers check the stamp — without calling the central desk. The stamp cannot be forged because only the venue knows the secret ink. No database lookup needed to verify who you are on every request.

JSON Web Tokens (JWTs) are the standard stateless authentication mechanism for REST APIs. A JWT encodes a payload (user ID, roles, expiration) into a base64 string, signed with a server-held secret via HMAC-SHA256 or RS256. Verification requires only the secret — not a database roundtrip — making JWT validation extremely fast and horizontally scalable.

Full JWT authentication implementation PYTHON 
from jose import JWTError, jwt
from passlib.context import CryptContext
from datetime import datetime, timedelta, timezone
from fastapi.security import OAuth2PasswordBearer, OAuth2PasswordRequestForm

SECRET_KEY = "use-a-256-bit-random-secret-from-env"
ALGORITHM = "HS256"
ACCESS_EXPIRE_MINUTES = 30
REFRESH_EXPIRE_DAYS = 7

pwd_ctx = CryptContext(schemes=["bcrypt"], deprecated="auto")
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="auth/login")

def hash_password(plain: str) -> str:
    return pwd_ctx.hash(plain)

def verify_password(plain: str, hashed: str) -> bool:
    return pwd_ctx.verify(plain, hashed)

def create_access_token(user_id: str, role: str) -> str:
    payload = {
        "sub": user_id,
        "role": role,
        "exp": datetime.now(timezone.utc) + timedelta(minutes=ACCESS_EXPIRE_MINUTES),
        "type": "access",
    }
    return jwt.encode(payload, SECRET_KEY, algorithm=ALGORITHM)

def decode_access_token(token: str) -> dict:
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        if payload.get("type") != "access":
            raise HTTPException(status_code=401, detail="Invalid token type")
        return payload
    except JWTError:
        raise HTTPException(status_code=401, detail="Invalid or expired token")

# Login endpoint
@router.post("/login")
async def login(
    form: OAuth2PasswordRequestForm = Depends(),
    db = Depends(get_db),
):
    user = await db.users.find_one({"email": form.username})
    if not user or not verify_password(form.password, user["password_hash"]):
        raise HTTPException(status_code=401, detail="Invalid credentials")
    token = create_access_token(str(user["_id"]), user.get("role", "user"))
    return {"access_token": token, "token_type": "bearer"}
Security Critical

Never store SECRET_KEY in code. Load it from environment variables via pydantic-settings. For production systems handling sensitive data, rotate to RS256 (asymmetric signing) and implement refresh token rotation with a Redis-backed revocation list.

MongoDB Integration — Motor Async Driver

The Simple Idea

MongoDB stores data as JSON-like documents instead of rigid table rows. Think of it as a filing cabinet where each folder can hold any kind of paper — some with 3 fields, others with 30 — and you can search across all of them simultaneously. Motor is the async version of the MongoDB driver, meaning queries don't freeze your server while they wait for the database to respond.

Motor is the official asynchronous Python MongoDB driver, wrapping PyMongo with asyncio-native coroutine support. All database operations become await-able, integrating natively into FastAPI's async event loop without thread pool overhead. MongoDB Atlas provides a managed cluster with built-in connection pooling, vector search, and global distribution.

database.py — Motor connection with connection pooling PYTHON 
from motor.motor_asyncio import AsyncIOMotorClient, AsyncIOMotorDatabase
from pydantic_settings import BaseSettings
from functools import lru_cache

class Settings(BaseSettings):
    MONGODB_URI: str = "mongodb://localhost:27017"
    DB_NAME: str = "myapp"
    # Connection pool: min 5, max 50 connections
    MONGO_MIN_POOL: int = 5
    MONGO_MAX_POOL: int = 50

    class Config:
        env_file = ".env"

@lru_cache
def get_settings() -> Settings:
    return Settings()

_client: AsyncIOMotorClient | None = None

async def connect_db():
    global _client
    cfg = get_settings()
    _client = AsyncIOMotorClient(
        cfg.MONGODB_URI,
        minPoolSize=cfg.MONGO_MIN_POOL,
        maxPoolSize=cfg.MONGO_MAX_POOL,
        serverSelectionTimeoutMS=5000,
    )
    # Verify connection on startup
    await _client.admin.command("ping")

async def close_db():
    if _client:
        _client.close()

def get_database() -> AsyncIOMotorDatabase:
    return _client[get_settings().DB_NAME]

# Register lifecycle hooks in main.py
@app.on_event("startup")
async def startup():
    await connect_db()

@app.on_event("shutdown")
async def shutdown():
    await close_db()

MongoDB's document model maps naturally to Pydantic models and JSON APIs. The critical data type to handle correctly is ObjectId — MongoDB's primary key type. It is a BSON type that doesn't serialize to JSON automatically. The production pattern wraps it to return string representations in API responses.

ObjectId handling — BSON to JSON bridge PYTHON 
from bson import ObjectId
from pydantic import field_serializer, model_validator
from typing import Any

def serialize_doc(doc: dict | None) -> dict | None:
    """Convert MongoDB document _id to string id."""
    if doc is None:
        return None
    doc["id"] = str(doc.pop("_id"))
    return doc

# Pydantic model with custom ObjectId field
class PyObjectId(str):
    @classmethod
    def __get_validators__(cls):
        yield cls.validate

    @classmethod
    def validate(cls, v: Any) -> str:
        if isinstance(v, ObjectId):
            return str(v)
        if not ObjectId.is_valid(v):
            raise ValueError("Invalid ObjectId")
        return str(v)

CRUD APIs with MongoDB

The Simple Idea

CRUD is the backbone of every API: Create a record, Read it back, Update it, Delete it. With MongoDB and Motor, all four operations are single async function calls. The challenge is doing them correctly — handling missing documents, validating ObjectIds, and never leaking database errors to clients.

Full async CRUD — production-grade user service PYTHON 
from fastapi import APIRouter, Depends, HTTPException
from motor.motor_asyncio import AsyncIOMotorDatabase
from bson import ObjectId

router = APIRouter(prefix="/users", tags=["Users"])

# CREATE
@router.post("/", response_model=UserResponse, status_code=201)
async def create_user(payload: UserCreate, db = Depends(get_db)):
    if await db.users.find_one({"email": payload.email}):
        raise HTTPException(409, "Email already registered")
    doc = payload.model_dump()
    doc["password_hash"] = hash_password(doc.pop("password"))
    doc["created_at"] = datetime.now(timezone.utc)
    result = await db.users.insert_one(doc)
    created = await db.users.find_one({"_id": result.inserted_id})
    return serialize_doc(created)

# READ (single)
@router.get("/{user_id}", response_model=UserResponse)
async def get_user(user_id: str, db = Depends(get_db)):
    if not ObjectId.is_valid(user_id):
        raise HTTPException(400, "Invalid user ID format")
    user = await db.users.find_one({"_id": ObjectId(user_id)})
    if not user:
        raise HTTPException(404, "User not found")
    return serialize_doc(user)

# READ (paginated list)
@router.get("/", response_model=list[UserResponse])
async def list_users(
    page: int = 1, limit: int = 20, db = Depends(get_db)
):
    skip = (page - 1) * limit
    cursor = db.users.find({}).skip(skip).limit(limit).sort("created_at", -1)
    users = await cursor.to_list(length=limit)
    return [serialize_doc(u) for u in users]

# UPDATE (partial)
@router.patch("/{user_id}", response_model=UserResponse)
async def update_user(
    user_id: str, payload: UserPatch, db = Depends(get_db)
):
    updates = payload.model_dump(exclude_unset=True)
    if not updates:
        raise HTTPException(400, "No fields to update")
    result = await db.users.find_one_and_update(
        {"_id": ObjectId(user_id)},
        {"$set": updates},
        return_document=True,
    )
    if not result:
        raise HTTPException(404, "User not found")
    return serialize_doc(result)

# DELETE
@router.delete("/{user_id}", status_code=204)
async def delete_user(
    user_id: str,
    db = Depends(get_db),
    _: dict = Depends(require_admin),
):
    result = await db.users.delete_one({"_id": ObjectId(user_id)})
    if result.deleted_count == 0:
        raise HTTPException(404, "User not found")

Middleware & Request Lifecycle

The Simple Idea

Middleware is a security checkpoint and logging booth that every request passes through before it reaches your code — and every response passes through on the way out. Add CORS rules, request IDs, response timing, and access logs in middleware once, and they apply to every single endpoint automatically.

Production middleware stack — main.py PYTHON 
from fastapi import FastAPI, Request
from fastapi.middleware.cors import CORSMiddleware
from starlette.middleware.base import BaseHTTPMiddleware
import uuid, time, logging

app = FastAPI(title="My API", version="1.0.0")

# 1. CORS — must be first in the stack
app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://myapp.com"],  # never * in production
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

# 2. Request ID + timing middleware
class RequestContextMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        request_id = str(uuid.uuid4())
        request.state.request_id = request_id
        start = time.perf_counter()

        response = await call_next(request)

        duration_ms = (time.perf_counter() - start) * 1000
        response.headers["X-Request-ID"] = request_id
        response.headers["X-Response-Time"] = f"{duration_ms:.2f}ms"

        logging.info(
            f"{request.method} {request.url.path} "
            f"{response.status_code} {duration_ms:.0f}ms [{request_id}]"
        )
        return response

app.add_middleware(RequestContextMiddleware)
IN

Incoming Request

HTTP request arrives at Uvicorn

Raw TCP connection accepted by the Uvicorn event loop, parsed into an ASGI scope, and forwarded up the middleware stack.

MW

Middleware Stack (descending)

CORS → Request ID → Logging → Auth pre-check

Each middleware wraps the next via await call_next(request). Middlewares execute in registration order on the way in, and in reverse on the way out.

RT

Router + Path Matching

FastAPI matches the URL to a handler function

Starlette's routing tree matches the request method and path. Path parameters are extracted and type-coerced.

DI

Dependency Resolution

DB session, auth token, config — all injected

FastAPI walks the dependency graph, resolving and caching all declared Depends() before calling the route function.

FN

Route Handler Execution

Your business logic runs here

The async def handler runs. Any await calls yield control to the event loop, allowing other requests to proceed concurrently.

OUT

Pydantic Serialization → Response

Return value validated, serialized, and sent to client

The return value is passed through the response_model serializer, converted to JSON, and streamed back through the middleware stack in reverse order before reaching the client.

WebSockets & Realtime Systems

The Simple Idea

HTTP is like sending letters — one question, one answer, connection closes. WebSockets are like a phone call — connection stays open, both sides can talk anytime. Use WebSockets for chat, live dashboards, multiplayer games, or any feature where the server needs to push data to the client without waiting for a request.

WebSocket with connection manager — broadcast chat room PYTHON 
from fastapi import WebSocket, WebSocketDisconnect
from typing import Dict

class ConnectionManager:
    def __init__(self):
        # room_id → set of active WebSocket connections
        self.rooms: Dict[str, set[WebSocket]] = {}

    async def connect(self, ws: WebSocket, room: str):
        await ws.accept()
        self.rooms.setdefault(room, set()).add(ws)

    def disconnect(self, ws: WebSocket, room: str):
        self.rooms.get(room, set()).discard(ws)

    async def broadcast(self, room: str, message: str, sender: WebSocket):
        dead = set()
        for ws in self.rooms.get(room, set()):
            if ws is sender: continue
            try:
                await ws.send_text(message)
            except:
                dead.add(ws)
        self.rooms[room] -= dead

manager = ConnectionManager()

@app.websocket("/ws/{room_id}")
async def websocket_endpoint(ws: WebSocket, room_id: str):
    await manager.connect(ws, room_id)
    try:
        while True:
            data = await ws.receive_text()
            await manager.broadcast(room_id, data, sender=ws)
    except WebSocketDisconnect:
        manager.disconnect(ws, room_id)

For production multi-server deployments, replace the in-memory ConnectionManager with a Redis Pub/Sub backend — each server publishes messages to a Redis channel, and all servers subscribe to forward them to their local WebSocket connections. This enables horizontal scaling of realtime features.

Background Tasks

The Simple Idea

When someone signs up, you want to respond instantly — not make them wait while you send a welcome email. Background tasks let you say "done!" immediately and do the slow work after the response is already on its way. For heavy jobs, use Celery with Redis as a proper task queue.

FastAPI BackgroundTasks + Celery task queue PYTHON 
from fastapi import BackgroundTasks

# Built-in: runs after response is sent, in same process
async def send_welcome_email(email: str, username: str):
    # slow SMTP operation — doesn't block the client

Error Handling & Exception Management

The Simple Idea

Good error handling means two things: clients always get a clean, consistent error shape — never a Python traceback — and your team always gets a detailed log. You control what the outside world sees and what your monitoring systems capture.

Global exception handlers — consistent error envelope PYTHON 
from fastapi import Request
from fastapi.responses import JSONResponse
from fastapi.exceptions import RequestValidationError
from starlette.exceptions import HTTPException

# Standardized error envelope for all responses
def error_response(status: int, code: str, message: str, detail=None):
    return JSONResponse(
        status_code=status,
        content={
            "error": {"code": code, "message": message, "detail": detail}
        },
    )

@app.exception_handler(HTTPException)
async def http_handler(request: Request, exc: HTTPException):
    return error_response(exc.status_code, "HTTP_ERROR", exc.detail)

@app.exception_handler(RequestValidationError)
async def validation_handler(request: Request, exc: RequestValidationError):
    errors = [
        {"field": ".".join(str(l) for l in e["loc"]), "msg": e["msg"]}
        for e in exc.errors()
    ]
    return error_response(422, "VALIDATION_ERROR", "Invalid request data", errors)

@app.exception_handler(Exception)
async def generic_handler(request: Request, exc: Exception):
    logging.exception("Unhandled error", exc_info=exc)  # full traceback in logs
    return error_response(500, "INTERNAL_ERROR", "An unexpected error occurred")

Caching with Redis

The Simple Idea

Caching is a photocopy machine for expensive answers. Instead of recalculating or re-querying the database every time someone asks the same question, you keep the last answer on a sticky note (Redis). Next time — answer from the sticky note. 1ms instead of 50ms.

Async Redis cache decorator pattern PYTHON 
import redis.asyncio as aioredis
import json
from functools import wraps

redis_client = aioredis.from_url("redis://localhost:6379", decode_responses=True)

def cache(ttl: int = 300, key_prefix: str = ""):
    """Async cache decorator with configurable TTL."""
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, **kwargs):
            cache_key = f"{key_prefix}:{func.__name__}:{args}:{kwargs}"
            cached = await redis_client.get(cache_key)
            if cached:
                return json.loads(cached)
            result = await func(*args, **kwargs)
            await redis_client.setex(cache_key, ttl, json.dumps(result, default=str))
            return result
        return wrapper
    return decorator

# Usage — cache for 5 minutes
@cache(ttl=300, key_prefix="user")
async def get_user_profile(user_id: str) -> dict:
    return await db.users.find_one({"_id": ObjectId(user_id)})

Rate Limiting & API Versioning

The Simple Idea

Rate limiting is a turnstile — it allows a maximum number of requests per user per minute, protecting your system from abuse and runaway clients. API versioning lets you ship breaking changes by routing /v1 clients to old logic and /v2 clients to new logic — without breaking anyone.

Rate limiting middleware + API versioning pattern PYTHON 
# Rate limiting with Redis sliding window
from fastapi import Request, HTTPException

async def rate_limit(request: Request, limit: int = 60, window: int = 60):
    key = f"rate:{request.client.host}:{request.url.path}"
    count = await redis_client.incr(key)
    if count == 1:
        await redis_client.expire(key, window)
    if count > limit:
        raise HTTPException(status_code=429, detail="Rate limit exceeded")

from fastapi import FastAPI

app.include_router(v1_router, prefix="/v1")
app.include_router(v2_router, prefix="/v2")

Production Architecture

The Simple Idea

A production system is layered defense and distributed load. Traffic flows from the internet through a load balancer that distributes it across multiple API instances, each of which checks Redis for cached answers before hitting MongoDB. Background workers handle slow tasks off the critical path. Every layer has redundancy.

Full Production Stack — Traffic Flow
Client (Browser / Mobile / API Consumer)
↓
Cloudflare CDN / WAF — DDoS protection, TLS termination
↓
NGINX Load Balancer — upstream health checks, SSL
↓
Gunicorn Process Manager (4 workers per instance)
↓
Uvicorn Worker × 4 → FastAPI Application
↓
Redis (cache layer) ← hit or miss → MongoDB Atlas (primary store)
↓
Celery Workers — email, exports, scheduled jobs
↓
Datadog / Sentry — observability, error tracking

The Gunicorn + Uvicorn combination is the production standard for FastAPI. Gunicorn manages worker processes and handles OS-level process supervision — restarting crashed workers, graceful reloads. Each worker is a UvicornWorker instance, running an async event loop. The recommended worker count is (2 × CPU cores) + 1.

Scaling FastAPI Systems

The Simple Idea

Scaling means more capacity. Vertical scaling adds more power to one machine. Horizontal scaling adds more machines. FastAPI is built to scale horizontally — spin up ten identical containers behind a load balancer, and you handle ten times the traffic. The key: make each instance stateless so any request can go to any server.

Stateless Design

Any server can handle any request

Store sessions in Redis, not in-process memory. JWT authentication is inherently stateless — no session store needed. Connection state for WebSockets requires Redis Pub/Sub for multi-node broadcast.

Async Concurrency

One worker handles thousands of I/O requests

A single Uvicorn worker can handle 1,000+ concurrent connections for I/O-bound workloads. Four workers on a 2-core instance can sustain 4,000+ concurrent connections — comparable to Node.js throughput.

MongoDB Connection Pooling

Reuse connections, don't open new ones per request

Motor maintains a connection pool (default: 100 connections per client). The Motor client must be instantiated once at startup and shared across all requests — never create a new client per request.

Database Indexing

Indexes turn 500ms queries into 1ms queries

Create compound indexes on MongoDB collections for every common query pattern. db.users.create_index([("email", 1)], unique=True). Missing indexes are the single largest source of database performance degradation at scale.

Best Practices

Architecture

Separation of Concerns

Routes, services, data — three distinct layers

Route handlers orchestrate, not execute. Business logic belongs in service functions. Database queries belong in repository functions. Never write MongoDB queries directly in route handlers. This makes each layer independently testable.

Performance

Never Block the Event Loop

Use async drivers for everything I/O

Replace synchronous libraries: requests → httpx; pymongo → motor; redis-py → redis.asyncio; open() → aiofiles. A single blocking call can stall all concurrent requests.

Security

Environment-Driven Config

Secrets never in code, always in env vars

Use pydantic-settings BaseSettings for all configuration. Load SECRET_KEY, MONGODB_URI, and API keys exclusively from environment variables. Use .env locally, secrets managers (AWS Secrets Manager, Vault) in production.

Production

Structured Logging

JSON logs with request IDs for correlation

Use structlog or configure Python logging with a JSON formatter. Always include request_id, user_id, and latency in log records. This enables distributed trace correlation in tools like Datadog, Grafana, or CloudWatch.

Advanced Backend Patterns

The Simple Idea

Once the basics work, senior engineers apply patterns that make large systems maintainable: repositories that hide MongoDB details, event buses that decouple components, circuit breakers that fail gracefully, and health endpoints that let infrastructure verify the app is operational before routing traffic to it.

Repository pattern + health check endpoint PYTHON 
# Repository pattern — abstract MongoDB away from business logic
class UserRepository:
    def __init__(self, db: AsyncIOMotorDatabase):
        self.collection = db.users

    async def find_by_email(self, email: str) -> dict | None:
        return await self.collection.find_one({"email": email})

    async def find_by_id(self, user_id: str) -> dict | None:
        return await self.collection.find_one({"_id": ObjectId(user_id)})

    async def create(self, data: dict) -> dict:
        result = await self.collection.insert_one(data)
        return await self.find_by_id(str(result.inserted_id))

# Health check — Kubernetes liveness/readiness probes
@app.get("/health", tags=["System"])
async def health_check(db = Depends(get_db)):
    checks = {}
    try:
        await db.command("ping")
        checks["mongodb"] = "ok"
    except Exception:
        checks["mongodb"] = "error"
    try:
        await redis_client.ping()
        checks["redis"] = "ok"
    except Exception:
        checks["redis"] = "error"

    healthy = all(v == "ok" for v in checks.values())
    return JSONResponse(
        status_code=200 if healthy else 503,
        content={"status": "healthy" if healthy else "degraded", "checks": checks},
    )
